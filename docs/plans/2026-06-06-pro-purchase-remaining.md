# Pro-покупки: оставшиеся работы

Дата: 2026-06-06

## Контекст
Базовый флоу Pro уже реализован и работает:
- Покупка `com.corvin.pro` ($0.99, non-consumable) — `Shared/Core/ProManager.swift` (StoreKit 2 + SK1 fallback для macOS 11).
- Restore через `AppStore.sync()` + `Transaction.currentEntitlements`.
- Пейвол `Shared/UI/ProPaywallView.swift`, локализация EN/RU.
- iOS: альтернативная иконка `ProAppIcon`, пикер в настройках, гейтинг по `isPro` — готово.

Решения по развилкам (зафиксированы 2026-06-06):
1. **Идентификация плательщика** — полагаться на Apple, бэкенд НЕ делаем.
2. **Благодарность** — экран-момент сразу после успешной покупки + шлифовка текстов.
3. **Иконка** — фича только для iOS; на macOS поправить вводящий в заблуждение текст пейвола.

- [ ] **macOS ProSettingsView** (`macOS/UI/Settings/SettingsView.swift`): убедиться, что благодарность видна и при покупке из настроек (сейчас показывает `star.fill` + `pro.thankYou` по состоянию — связать с тем же сигналом).

## Блок C — Идентификация плательщика (без бэкенда)

Apple привязывает покупку к Apple ID и сам хранит/проверяет entitlement. Приложение НЕ получает имя/почту покупателя (политика приватности). App Store Connect показывает только обезличенную агрегированную статистику. Восстановление между устройствами одного Apple ID работает через restore без своей серверной логики.

Кодовых задач почти нет:
- [ ] Убедиться, что кнопка restore доступна и работает на обеих платформах (iOS-пейвол + macOS `ProSettingsView` — есть; проверить доступность на первом запуске/онбординге).
- [ ] Зафиксировать решение в коде/README: идентификация и синхронизация — на стороне Apple (через Apple ID + restore).

> Если в будущем понадобится знать «кто именно купил» (именная благодарность, кросс-платформенная синхронизация вне Apple ID, аналитика) — потребуется свой аккаунт-бэкенд + `transaction.appAccountToken` + серверная валидация (App Store Server API). Сейчас вне объёма.

## Блок D — Фикс: смена языка не применяется

**Симптом:** в настройках («Язык») выбор English ничего не меняет — все надписи остаются на русском до перезапуска (а фактически и после выбора ничего не происходит).

**Корневая причина** (`Shared/Core/LocalizationManager.swift`):
- `currentLanguage.didSet` корректно зовёт `Bundle.setLanguage()` — подмена bundle на нужный `*.lproj` работает.
- НО `String.localized` (extension, ~строки 214–224) — обычное вычисляемое свойство: читает `Bundle.localizedBundle` в момент построения `body` и возвращает простую `String`, не реагирующую на изменения.
- Большинство вью НЕ наблюдают `LocalizationManager`. Ярлыки сайдбара берутся из `SettingsTab.label` (`macOS/UI/Settings/SettingsView.swift`, enum, не View) — `.localized` там вычисляется один раз и не пересчитывается.
- Итог: дерево SwiftUI не перестраивается, язык «применяется» только при следующем запуске (когда `Bundle.setLanguage` отрабатывает на старте).

**Задачи:**
- [ ] Сделать смену языка реактивной: каждый корневой SwiftUI-view, отображаемый через `NSHostingView`, должен `@ObservedObject`'ить `LocalizationManager.shared` и нести `.id(localization.currentLanguage)` на корневом контейнере — тогда при смене языка всё поддерево пересобирается и все `.localized` (в т.ч. enum-ярлыки `SettingsTab.label`) пересчитываются против нового bundle.
  - Применить ко всем корням: Settings (`SettingsView`), окно Истории, Онбординг, ModelManager-окно, FloatingIndicator.
- [ ] Проверить, что в собранном бандле `en.lproj`/`ru.lproj` реально находятся по пути `Bundle.main.path(forResource: <lang>, ofType: "lproj")` (XcodeGen копирует `Shared/Resources` папкой — убедиться, что lproj лежат в корне бандла, а не вложенной папкой; иначе `setLanguage` тихо падает в `Bundle.main`/системный язык).
- [ ] AppKit-части (меню статус-бара `StatusBarController`, любые `NSMenu`) реактивность SwiftUI не покрывает — перестраивать их по уведомлению о смене языка (напр. `NotificationCenter` из `LocalizationManager.currentLanguage.didSet`).
- [ ] Когда живое переключение заработает — убрать/смягчить подпись `settings.language.restartRequired` («может потребоваться перезапуск»).
- [ ] `AppLanguage.displayName` для `.system` тоже использует `.localized` — проверить, что в реактивном дереве он обновляется корректно.

**Проверка:** переключить En↔Ru в настройках без перезапуска — сайдбар, активная вкладка, меню статус-бара и онбординг должны мгновенно сменить язык.

**Статус (2026-06-06):** механизм реактивности реализован — окно настроек (всё, что использует `.localized`) теперь переключается вживую без перезапуска. Сборки macOS+iOS зелёные; `en.lproj/ru.lproj` подтверждены в корне `Corvin.app/Contents/Resources/` (проблема была не в пути, а в реактивности).

**Follow-up (отдельная задача i18n):** ряд экранов всё ещё содержит ХАРДКОД русских строк (не `.localized`), поэтому они не переключатся, пока их не локализуют:
- [ ] Меню статус-бара (`StatusBarController.buildMenu()` — «готов», «запись...», «Последние записи:» и т.д.). Инфраструктура перестроения по уведомлению уже на месте — нужны только ключи.
- [ ] `GeneralSettingsView` — кнопки «Экспорт логов» / «Очистить логи» и подпись под ними.
- [ ] Заголовки окон в `AppDelegate` («Настройки Corvin», окно истории и т.п. — задаются один раз через `NSWindow.title`, реактивностью SwiftUI не покрыты; обновлять по `.appLanguageChanged`).
- [ ] Тексты в `OnboardingView` / `HistoryWindowView`, если там остались литералы.

## Блок E — Чистка (опционально) — ОТЛОЖЕНО

Расследование (2026-06-06) подтвердило, что `LicenseKit` действительно вестигиальный:
- KeyboardKit 10.4.1 `Package.swift` зависит только от MockingKit — НЕ от LicenseKit.
- В `project.yml` LicenseKit нет в `packages:` и ни в одной target-зависимости — только в строке embed-скрипта (`project.yml:81`, связка `KeyboardKit.framework LicenseKit.framework`).
- Ссылок в коде нет. Резолвится лишь потому, что остался в уже сгенерированном (не трекаемом git) `Corvin.xcodeproj` + `Package.resolved`.

**Почему отложено:** безопасное удаление требует `make project` (регенерация `.xcodeproj`) + сборки/запуска iOS-клавиатуры на устройстве для проверки, что embedding не сломался (исторически хрупкий — см. [[keyboardkit-embedding]]). Регенерация рискует задеть уже изменённые в рабочем дереве `Info.plist`. Профита по функциональности — ноль. Делать отдельной сфокусированной задачей вне этой ветки.

- [ ] (later) Убрать `LicenseKit.framework` из embed-loop в `project.yml:81`, `make project`, собрать macOS + iOS, проверить клавиатуру на устройстве, закоммитить.
